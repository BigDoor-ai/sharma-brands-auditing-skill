# AGENTS.md — Sharma Brands-Inspired Website Auditing Skill

Use these instructions whenever the user asks you to audit, build, redesign, improve, or review a website, landing page, SaaS page, ecommerce page, product page, lead-generation page, or marketing funnel.

This is an independent conversion-audit framework inspired by publicly available teachings from Nik Sharma and Sharma Brands. It is not affiliated with, endorsed by, sponsored by, certified by, or officially approved by Nik Sharma or Sharma Brands.

## Core mantra

A cold visitor should understand within 3 seconds:

1. What is being sold.
2. Who it is for.
3. Why it matters.
4. Why it is credible.
5. What action to take next.

Prioritize clarity, buyer education, trust, low friction, real proof, mobile-first UX, and specific CTAs.

## Strict truth policy

Never invent:

- Testimonials
- Client logos
- Case studies
- Pricing
- Guarantees
- Performance metrics
- Customer names
- Certifications
- Legal/compliance claims
- Delivery timelines

If proof is missing, use placeholders like:

- [Add verified client logo]
- [Add customer testimonial]
- [Add real setup timeline]
- [Add pricing details]
- [Add case study metric]
- [Add product screenshot]
- [Add compliance/certification details]

Do not present placeholders as real claims.

## 3-second clarity test

The hero area must answer:

- What is this?
- Who is it for?
- What result does it create?
- Why should the visitor keep scrolling?
- What should the visitor do next?

Flag vague headlines, abstract slogans, clever-but-unclear copy, missing visuals, missing CTAs, and missing proof.

## Hero requirements

A strong hero should include:

- Clear headline
- Outcome-focused subheadline
- Product/service/workflow visual
- Primary CTA
- Trust signal near CTA
- Optional anxiety-reducing secondary CTA

Avoid vague CTAs like:

- Submit
- Learn more
- Get started
- Click here

Prefer specific CTAs like:

- Book a 15-minute demo
- Request inventory sheet
- Start WhatsApp setup
- Get wholesale pricing
- See product catalog
- Try the scanner
- Get my audit

## Default conversion page structure

Unless the user gives a different structure, recommend or build:

1. Hero
2. Trust bar
3. Problem
4. Solution workflow
5. Benefits
6. Product/service demo
7. Use cases
8. Proof
9. Pricing/offer
10. FAQs
11. Final CTA
12. Footer

## Buyer questions every page must answer

Every page should directly answer:

1. What is the product or service?
2. Who is it for?
3. What problem does it solve?
4. What outcome does it create?
5. Why should the buyer trust it?
6. What happens next after the CTA?

## Copy rules

Write copy that is:

- Simple
- Specific
- Buyer-language-driven
- Outcome-focused
- Proof-backed
- Scannable
- Easy to understand on mobile

Avoid:

- Vague startup language
- Unsupported superlatives
- Overly clever headlines
- Long founder story before clarity
- Abstract benefits
- Feature dumping
- Empty adjectives like seamless, revolutionary, cutting-edge, unless supported

## Proof rules

Proof should appear early and throughout the page.

Strong proof can include:

- Client logos
- Real testimonials
- Case studies
- Screenshots
- Product demo outputs
- Usage numbers
- Before/after workflows
- Certifications
- Press mentions

Use proof to answer objections, not merely decorate the page.

## Friction checks

Flag these conversion blockers:

- Vague headline
- No clear buyer
- No obvious CTA
- Too many competing CTAs
- Missing proof
- Missing pricing direction
- Too many form fields
- Weak mobile readability
- Important information hidden on other pages
- Clickable-looking elements that do nothing
- No explanation of what happens after form submission
- No FAQ for obvious objections
- Slow or heavy page experience

## Forms

Forms should ask only what is needed for the next step.

Good B2B demo form:

- Name
- Company
- Email or phone
- Requirement/use case

Always explain what happens after submission.

## Mobile-first rules

Assume the first visit is mobile.

Check:

- Readable hero without zooming
- CTA visible early
- Large tap targets
- Short paragraphs
- No horizontal scroll
- Fast-loading visuals
- Mobile-friendly forms
- Sticky CTA where appropriate

## When editing code

If asked to improve a codebase:

1. Audit first.
2. Identify files/components likely needing changes.
3. Propose prioritized fixes.
4. Edit code only after stating the plan, unless the user asked for direct implementation.
5. Use semantic HTML.
6. Build mobile-first responsive sections.
7. Add reusable CTA/proof/FAQ components when useful.
8. Add SEO metadata and Open Graph metadata when missing.
9. Add analytics placeholders when useful.
10. Keep content easy to edit.

## Final standard

The site is successful only if visitors understand the offer, trust it, and can take the next step without confusion.
